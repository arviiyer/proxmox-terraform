## What This Project Does

This repository now contains a single Go web application:

**forge** (`portal/`) â€” an EC2-like interface for provisioning homelab VMs on Proxmox using Terraform. Users submit a launch form, forge writes a Terraform vars file, runs `terraform apply` against `infra/`, and exposes VM lifecycle controls from the UI.

## Running And Building

```bash
cd portal && go run main.go
cd portal && go build -o forge .
```

Required env vars:
```bash
PVE_ENDPOINT="https://your-proxmox-host"
PVE_API_TOKEN="user@pam!tokenid=secret"
SSH_PUBLIC_KEY="ssh-ed25519 ..."
```

Optional env var:
```bash
SSH_NODE_KEY_FILE="/home/user/.ssh/id_ed25519"
```

The private key at `SSH_NODE_KEY_FILE` is used by the bpg/proxmox Terraform provider to SSH into Proxmox nodes as `root` when writing snippet files for cloud-init user data.

No external Go dependencies are used; the app is stdlib-only. Terraform requires `>= 1.5.0` with the `bpg/proxmox` provider.

## Architecture

### `portal/`

- `main.go`: HTTP routes, form parsing, merge logic, async launch/destroy jobs, protection toggles, and server-rendered templates.
- `terraform/runner.go`: wraps `terraform init`, `apply`, `apply -destroy`, `show -json`, `output -json`, and `refresh-only` using `os/exec`.
- `proxmox/client.go`: thin Proxmox REST client for start, stop, status, and protection operations that do not need a Terraform apply.
- `config.json`: allowlisted nodes, templates, instance types, and runtime defaults.
- `protected.json`: runtime-managed protected VM list; do not hand-edit it while forge is running.
- `web/templates/`: HTML templates for the dashboard, results, and job pages.

### `infra/`

- `main.tf`: Proxmox provider setup, flavor locals, VM resource definition, cloud-init support, and `stop_on_destroy = true`.
- `variables.tf`: all configurable Terraform inputs.
- `outputs.tf`: current instance outputs filtered to usable IPs.
- `portal.auto.tfvars.json`: runtime-generated; do not edit manually.

## Key Behaviors

- Templates and instance types are validated against `config.json` allowlists before any action.
- VMID range is `100..999999`; launch count is `1..50`.
- Existing VMs are read from `terraform show -json`, not cached outputs, before every launch.
- New launches merge with existing state so a new apply does not destroy prior VMs.
- A `terraform apply -refresh-only` pass runs after provisioning to pick up DHCP-assigned IPs.
- Destroy uses targeted `terraform apply -destroy -target=...` for single-VM deletion.
- Protected VMs require the literal confirmation phrase `destroy protected instance`.
- Start and stop use direct Proxmox API calls for speed.
- Stop uses ACPI shutdown with `forceStop=1` fallback.
- VM power states are fetched in parallel on page load.
- User data is optional. When present, forge uploads a snippet to Proxmox storage and passes its file ID into Terraform.
- Terraform state is local in `infra/terraform.tfstate`.

## Repo-Specific Gotchas

- Prefer `terraform show -json` over `terraform output -json` for current resource truth.
- Do not break the merge-before-apply behavior; it is what prevents unrelated VM destruction.
- Preserve `stop_on_destroy = true` in Terraform.
- Keep `portal.auto.tfvars.json` generated by the app, not hand-maintained.
- The app assumes Proxmox API TLS verification is skipped, matching Terraform provider config.
- If you change shared launch or destroy flow, verify both the UI handler path and Terraform runner behavior.
